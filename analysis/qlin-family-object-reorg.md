# q_lin reorg: constraint-band objects (the build-`r_i` side)

Status: **design note, not yet implemented** (2026-06-23). Companion to
`prover-optimization-investigation.md` (where the prove-time breakdown was
*measured*) and to `qlin-evenodd-fused-multiply.md` (the multiply side). This
note covers the **build-`r_i`** half of the q_lin fold: how the prover turns
claims into the per-row challenged-constraint coefficients (`chunk_rTA`), and a
reorganization that removes the per-row packet store, the lazy compile, and the
`_late_qlin_var` path ŌĆö while recovering measured overhead.

Prover-only: the verifier compiles independently, so nothing here changes
constraint IDs, the layout, or what the verifier checks.

## TL;DR

- The packet store is **per-row** (millions of near-identical objects;
  `matmul_compile` appends one `L2_IdentityScalar` per row of `y`). That bloats
  to ~45 GB at 48-layer scale, which forced **lazy per-op compile**
  (`_StreamingPackets`), which created **`_late_qlin_var`** (re-encode + patch),
  which created **two lifetime systems** (dataflow `last_use` + constraint
  completeness). All of it is downstream of the per-row representation.
- **Reorg:** store **one band object (`ConstraintBand`) per `(variable, family)`** ŌĆö
  effectively a reference to the claim plus its cid `base` ŌĆö and have it generate
  its dense per-slot contribution at fold time. Compile becomes a count pass
  (assign `base`) + a reverse index (`variable ŌåÆ [band]`), `O(claims ├Ś families)`,
  megabytes not gigabytes.
- **The sort drops out.** Each band reduces its own (regular) structure;
  cross-family overlap is a small `gl_add`. No `argsort`, no CSR, no segmented
  spmv.
- **CUDA contract:** each band yields a dense `(n_rows, ELL)` block (the
  existing `chunk_rTA` shape); bands accumulate with `gl_add`. Only one new
  primitive (`challenge_at`) and a field reduce for fan-out kinds.
- **Recovers measured overhead:** the `compile` bucket (~11%, ~37 s) and most of
  the `expand` overhead (part of the ~58 s) ŌĆö because building `r_i` is
  witness-independent and ~95% structurally repeated, yet is regenerated every
  chunk today. It does **not** touch `poly_mul` (the NTTs); that's the multiply
  note.

## 1. Abstraction: claim ŌåÆ family ŌåÆ constraint ŌåÆ band

Four distinct levels (the middle two are easy to conflate):

- **Claim** ŌĆö a tape op (e.g. `MatmulClaim`).
- **Constraint family** ŌĆö a *group* of constraints generated by one rule,
  occupying a contiguous cid-range and (often) sharing one challenge. Matmul LF1
  is "for every contraction index `i_k`, `y[i_k] = ╬Ż_j Žü[j]┬ĘB[ŌĆ”,j]`": one rule,
  `k` constraints in `cid Ōłł [base, base+k)`. The family is the *shared* unit ŌĆö
  several variables contribute to it (below) ŌĆö and is a **conceptual** lens, not a
  materialized object (see ┬¦11).
- **Constraint** ŌĆö one linear equation = one `cid` = one row of `A`.
- **Band** ŌĆö one *variable's* contribution to one family: the materialized
  implementation object (`ConstraintBand`). A single constraint spans several
  variables (the LF1 row for `i_k` touches `y[i_k]` *and* a row of `B`), so a
  constraint isn't localizable to one variable. What *is* localizable is "how
  variable `v`'s slots map into family F's cids, with what coefficients" ŌĆö that's
  the band (today's per-row packet, but instanced **once per variable** instead of
  once per row).

So one family yields *two* bands for LF1: `(y, LF1)` ŌåÆ identity coefficients, and
`(B, LF1)` ŌåÆ Freivalds coefficients, both pointing at the same `[base, base+k)` but
describing different sides. The bands are **individually owned per variable**; the
only thing they share is the cid `base` (assigned by the count pass). ┬¦11 covers
why this ŌĆö not a shared family-object ŌĆö is the right model.

## 2. Current flow and where it's wasteful

Compile (`_compile_with_chs`, `core.py:1642`) walks claims; each `compile_fn`
returns `(witness_row, packet)` pairs scattered into `per_row[r].append(pkt)`
(`core.py:1679-1680`). At 48-layer scale this per-row store is ~45 GB, so the
streaming prover compiles **lazily per op** (`_StreamingPackets`, `core.py:953`)
and patches constraints that compile after a variable's rows already emitted via
`_late_qlin_var` (`core.py:1969`; rationale at `core.py:968-975`).

The fold (`_compute_q_lin_inner_chunk`, `core.py:1139`) per chunk:

1. bin `per_row_packets` by kind, run each kind's expander (`packets.py`,
   `EXPANDERS` at `packets.py:68`) ŌåÆ flat `(target_idx, cid, coef)` triples;
2. `torch.argsort(target_idx)` ŌåÆ `bincount` ŌåÆ `row_ptr` (`core.py:1186-1192`) ŌĆö
   the **sort**, to group triples by target slot;
3. `gl_spmv_challenged` (`core.py:1193`) ŌĆö segmented sum, `challenge(cid)`
   inline ŌåÆ `chunk_rTA (n, ELL)`;
4. `_interpolate_to_kdeg` (`core.py:896`) ŌåÆ `poly_mul_batched`
   (`cuda_primitives.py:917`).

Measured shares (`prover-optimization-investigation.md` ┬¦1ŌĆō2): `expand Ōēł 58 s`,
`sort Ōēł 23 s`, `spmv Ōēł 10 s` inside `fold_qlin`; `compile Ōēł 11%`. The decisive
fact: **steps 1ŌĆō3 (`chunk_rTA`) are entirely witness-independent** ŌĆö they depend
only on the constraint layout and the challenges ŌĆö and are **~95% structurally
repeated** (6316 q_lin chunks ŌåÆ 322 distinct structures, `LIGERO_REUSE_HASH`).
Yet the code regenerates them from scratch every chunk, on top of per-row packet
object churn. That regeneration + churn is the recoverable overhead.

## 3. The reorg

### Compile = count pass + reverse index

- **Count pass** (already done, for `r_quad` sizing): assign each `(claim,
  family)` its cid `base` from the running constraint total (prefix sums of
  `n_added`). No per-row work.
- **Reverse index**: per variable, a list of **bands** (one per `(claim, role)`
  touching it). Compact: `O(claims ├Ś families)`.

Because this is tiny, it can be done **upfront** (no 45 GB store, no lazy
compile) ŌĆö which is the precondition for folding at emit with complete
constraints, so `_late_qlin_var` has no reason to exist.

### The band object

```python
class ConstraintBand:
    """One variable's contribution to one constraint family (a shared cid-group).
    Metadata only, created at the count pass; generates its dense per-slot block on
    demand. Owned by exactly one variable; the family it serves is shared (see ┬¦1,
    ┬¦11) and exists only as the cid numbering."""
    base: int            # cid base of the (shared) family this band serves
    var_row_start: int   # the variable's row_start (for row_off)
    L: int               # variable length (last-row clamp)
    # role-specific dims (k, n, H, stride, fan, ...) + coef source:
    coef: int | torch.Tensor   # scalar, or a small device tensor (neg_rho /
                               # coef_vec) computed ONCE from ch0 and cached here

    def contribution(self, row_lo, row_hi,
                     seed, label, ELL) -> torch.Tensor:
        """Dense (row_hi-row_lo, ELL) uint64: this band's challenged coefficients
        per slot, vectorized over the row range. No COO, no sort."""
        ...
```

The `role` (a subclass, or an enum + dispatch) replaces the `L2_*` kind ŌåÆ
`EXPANDERS` registry. Same per-role logic, relocated into `contribution`, and
instanced **once per variable** instead of once per row.

### Build `r_i` variable-stationary ŌĆö the sort drops out

```python
def build_r_eval(variable, row_lo, row_hi, seed, label, ELL):
    r_eval = None
    for band in variable.bands:                  # 1ŌĆō3 bands per variable
        c = band.contribution(row_lo, row_hi, seed, label, ELL)  # dense (n,ELL)
        r_eval = c if r_eval is None else gl_add(r_eval, c)
    return r_eval        # == today's chunk_rTA; feeds _interpolate_to_kdeg +
                         # poly_mul UNCHANGED
```

Why no sort: today's `argsort` exists only because all families' triples are
merged into one unordered COO list, so grouping-by-target must be rediscovered.
Keeping per-family structure makes the grouping explicit:

- **1:1 families** (identity, Freivalds): one cid per slot ŌåÆ one value per slot,
  written at `local_row┬ĘELL + slot`. No reduction.
- **fan-out families** (stride-one-to-many, transpose-o2m, causal-c2): one slot
  ŌåÆ a *contiguous* cid range ŌåÆ a `reshape (n, ELL, fan)` + field-sum over `fan`,
  using the family's known stride. A structured reduce, not a sort.
- **cross-family overlap** (a slot bound by 2+ families): the `gl_add` across the
  variable's bands. Fan-in is tiny (1ŌĆō3).

### Two concrete `contribution`s

1:1 ŌĆö `FreivaldsLF1B` (each `B`-slot ŌåÆ one cid):

```python
def contribution(self, row_lo, row_hi, seed, label, ELL):
    n = row_hi - row_lo
    rows = torch.arange(row_lo, row_hi, device="cuda").unsqueeze(1)
    slot = torch.arange(ELL, device="cuda").unsqueeze(0)
    flat = (rows - self.var_row_start) * ELL + slot            # (n,ELL) B-index
    valid = flat < (self.k * self.n)
    j, i_k = self._decode(flat)                                # role logic
    cids = self.base + i_k                                     # one cid per slot
    chal = challenge_at(seed, label, cids)                     # (n,ELL)  <- new
    head = i_k // self.K
    coef = self.neg_rho.index_select(0, (head*self.n + j).flatten()).view(n, ELL)
    return gl_mul(chal, coef) * valid
```

fan-out ŌĆö `StrideOneToManyScalar` (each slot ŌåÆ `stride` cids, summed into it):

```python
def contribution(self, row_lo, row_hi, seed, label, ELL):
    n = row_hi - row_lo
    rows = torch.arange(row_lo, row_hi, device="cuda").unsqueeze(1)
    slot = torch.arange(ELL, device="cuda").unsqueeze(0)
    flat = (rows - self.var_row_start) * ELL + slot
    valid = flat < self.L
    t = torch.arange(self.stride, device="cuda")
    cids = (self.base + flat * self.stride).unsqueeze(-1) + t   # (n,ELL,stride)
    chal = challenge_at(seed, label, cids)                      # (n,ELL,stride)
    summed = gl_sum(chal, dim=-1)                               # internal reduce
    return gl_mul(summed, self.coef) * valid
```

## 4. What's easiest for CUDA to consume

**Dense `(n_rows, ELL)` blocks accumulated with `gl_add`.** Reasons:

- **Zero downstream change.** `r_eval` is bit-identical in shape to today's
  `chunk_rTA`, so `_interpolate_to_kdeg` and `poly_mul_batched` are untouched.
- **All coalesced, regular ops** ŌĆö `arange`, broadcast, `index_select`,
  `gl_mul`, a field reduce, `gl_add`. None of `argsort` / irregular CSR gather /
  arbitrary scatter.
- **No field-safe atomics in the common case.** Full-coverage families
  (Freivalds on weights, identity copies ŌĆö the bulk) write a complete dense
  block and `gl_add`. Only *sparse* families (causal masks, partial last rows)
  need a field-scatter-add into their active slots.
- **Layout:** keep blocks **row-major `(n_rows, ELL)`** (slot contiguous within a
  row) so `challenge_at`/`gl_mul` are coalesced across `ELL` and the downstream
  iNTT reads rows contiguously ŌĆö same layout as today's `chunk_rTA`.

New CUDA pieces (small):

1. **`challenge_at(seed, label, cids) -> challenges`** ŌĆö the BLAKE3 challenge as
   a plain gather over an arbitrary cid tensor. It is `gl_spmv_challenged`'s
   inner hash with the segmented sum removed (`cuda_primitives.py:945`).
2. **A field reduce over the last axis** for fan-out (`gl_sum(x, dim=-1)`) ŌĆö a
   batched `gl_matvec`-with-ones, or a small dedicated reduce.

## 5. Role table (which kinds are full-coverage vs sparse)

| current `L2_*` kind | role | per-slot cids | coverage |
|---|---|---|---|
| `L2_IdentityScalar` | identity | 1 | full |
| `L2_PerSlotVector` | identity + vec coef | 1 | full |
| `L2_RowSumPerSlotVector` | strided cid + vec coef | 1 | full |
| `L2_FreivaldsLF1B` / `LF2A` / `LF3C` | Freivalds | 1 | full |
| `L2_StrideManyToOneScalar` | shared cid (`cid//stride`) | 1 | full |
| `L2_EmbedE` | token-gated identity | 1 | sparse (hits) |
| `L2_StrideOneToManyScalar` | fan-out (contiguous) | `stride` | full |
| `L2_TransposeO2MScalar` | fan-out (transposed) | `fan` | full |
| `L2_CausalFilteredIdScalar` | causal identity | 1 | sparse (triangular) |
| `L2_CausalFilteredC2Stride` | causal fan-out (ragged) | `i_qry+1` | sparse |
| `L2_RoPEX` / `RoPEXRot` | per-slot vec coef | 1 | full |

"sparse" = needs field-scatter-add (it doesn't touch every slot of every row);
everyone else is a dense `gl_add`.

## 6. What it removes, recovers, and does not touch

Removes: the per-row packet store (45 GB), lazy compile (`_StreamingPackets`),
`_late_qlin_var`, the dual lifetime systems, and the `argsort`/CSR sort.

Recovers (measured overhead): the `compile` bucket (~11%, ~37 s; per-row object
churn ŌåÆ per-family descriptors) and most of the `expand` overhead (part of the
~58 s; per-chunk re-binning, per-packet Python loops, `cat`, and regeneration of
the ~95%-repeated structure). The irreducible remainder of building `r_i` is the
nnz materialization + the challenge apply (Ōēł the measured `spmv Ōēł 10 s`).

Does **not** touch: `poly_mul` (the ~74 s of NTTs). That is the multiply note's
domain. The two compose: this note produces `r_eval` cheaply and densely; the
multiply note consumes it.

## 7. Risks / open items

- **Expander interface refactor** is the real work: rewrite each `L2_*`
  expander as a `ConstraintBand.contribution` reading from the object + a row
  range. Mechanical, per-role, but it is the bulk of the diff.
- **Per-claim coef caching**: compute challenge-derived coefs (`neg_rho`,
  `(╬▒ŌłÆv)`) **once** per family and cache on the object ŌĆö not per chunk, or you
  re-run `gl_neg(rho)` for every chunk of a variable's rows.
- **Chunk vs variable boundary**: prefer a variable-stationary fold (run a
  variable's families over its contiguous rows) so the family lookup aligns with
  the layout; the inner-chunk re-chunking (`QLinAccumulator`, `core.py:1099`) can
  stay underneath for the `prods` intermediate.
- **Confirm upfront compile is cheap once compacted.** Today the full compile is
  "~87 GB / minutes" at 32L/SEQ=1000; the working assumption is that the
  minutes are per-row tensor building and that per-family descriptors drop it to
  seconds. Verify with a count-pass timing before relying on upfront compile.

## 9. R3 control flow (sound) and the three test polynomials

The reorg's natural fold is **variable-stationary**, nested under the per-op
witness regeneration. The op must stay the *outer* driver: sound R3 recomputes
the witness, and an op's `COMPUTE_FN` produces its outputs from earlier ops'
values, so iteration follows dataflow order.

```
for op in ops:                       # outer: regenerate witness + aux, dataflow order
    compute op's witness + aux
    for v in op's variables:         # inner: the encode+fold unit
        encode v's rows ONCE -> f_v_coeffs (+ message/slack for the even half)
        for acc in [q_irs, q_lin]:   # accumulator list ŌĆö no per-test conditional
            acc.update(v, encode)
    p_0.fold(op's own quads)         # constraint-stationary, separate, uniform
    free v per last_use
```

The three tests in this loop:

- **q_irs** ŌĆö `q_irs += r_irs[v.rows] ┬Ę f_v_coeffs`: a coefficient-domain matvec,
  **no NTT of its own** (it rides on the encode's iNTT). Always applies (per row).
- **q_lin** ŌĆö gather `v`'s families (complete, since compile is upfront), build
  `r_eval` (per-family reduce + `gl_add`, no sort), even/odd multiply, accumulate.
  Variable-stationary. (Multiply: `qlin-evenodd-fused-multiply.md`.)
- **p_0** ŌĆö folded at the **op** level over the op's own quads (the 3-row "triple
  property" couples variables, so it can't live inside one variable's pass). It
  stays constraint-stationary and re-encodes its operands, exactly as today.

So the **variable** is the unit for the linear tests; the **op-grouped
constraint** is the unit for the quadratic test. One encode per variable feeds
both linear tests, and `_late_qlin_var` is gone because upfront compile makes
each variable's family list complete at its emit.

## 10. Keeping the implementation conditional-free

The pseudocode stays close to the code if the branch-prone spots are dissolved
structurally rather than with `if`:

- **Per-family-kind logic** ŌåÆ polymorphic `ConstraintBand.contribution()`
  (method dispatch; today's `EXPANDERS[kind]` dict is already this shape).
- **Full-coverage vs sparse families** ŌåÆ uniform contract: `contribution()`
  *always* returns a dense `(n_rows, ELL)` block (sparse kinds return zeros via
  the `* valid` mask ŌĆö a vectorized multiply, not control flow), and the caller
  *always* `gl_add`s. One accumulation path.
- **The linear accumulators** ŌåÆ an accumulator list (`for acc in accumulators:
  acc.update(...)`), the existing `_stream_phase` shape.
- **Even/odd** ŌåÆ a static index partition (even slice / odd slice), two fixed
  blocks that always run ŌĆö slicing, not branching.
- **p_0** ŌåÆ kept a separate, internally-uniform constraint-stationary pass; do
  **not** fuse it into the linear loop (that's where a cross-op `if` would live).

Residual guards are minimal: an optional `if not v.bands` short-circuit, and
`if op.quads` before the p_0 fold ŌĆö per op, not per row. Net, the reorg has
*fewer* branches than today's lazy-compile + late-path code.

## 11. Decisions and deferrals

- **Don't hold encoded values** (for now). The value form is held to `last_use`
  for dataflow anyway; holding the encoded form (`Ōēł K_DEG`-wide, Ōēł the value
  width since the ZK slack is small) is *additive* and Ōēł doubles the working-set
  memory. So the encode stays transient per variable (produce ŌåÆ fold ŌåÆ free).
  Revisit only with materially more RAM.
- **No op-local p_0 sharing** (for now). Reusing a variable's q_lin encode for an
  op-local quad is a real but modest win (p_0 is ~10% of prove) and reintroduces
  a cross-op conditional. Keep p_0 uniformly re-encoding; revisit after the
  bigger levers land.
- **Per-variable iteration, op-outer** ŌĆö the variable is the encode+fold unit;
  the op stays the outer driver (witness regeneration + dataflow order). Large
  variables sub-chunk their rows for the transform (bound the `prods`
  intermediate); the variable is still the logical unit.
- **Codeword reuse rejected** ŌĆö see `qlin-evenodd-fused-multiply.md` ┬¦7.
- **Bands individually owned per variable; the family is only shared numbering.**
  A linear constraint couples several variables (LF1's `[base, base+k)` binds both
  `y` and `B`), but the design does **not** materialize a shared family-object that
  variables attach to. Each variable owns its bands; the only shared state is the
  cid `base` from the count pass (deterministic and verifier-recomputable). The
  cross-variable coupling is recombined *for free* by the linear test's linearity ŌĆö
  `test = ╬Ż_v Ō¤©r_eval_v, w_vŌ¤® = ╬Ż_c r[c]┬Ęconstraint_c` ŌĆö so the per-variable
  `r_eval`s (built independently in `build_r_eval`) add up on shared cids: they
  share `r[c]` (same cid ŌåÆ same challenge) and the test sums over variables. A
  shared family-object would be redundant (the test already does the recombine) and
  would re-introduce the cross-variable fold-lifetime coordination this reorg
  deletes (a family spans variables folded at different times under streaming). So
  **"constraint family" stays a conceptual/spec unit** (how we *talk* about what the
  verifier checks); **`ConstraintBand` is the materialized per-variable object.**
